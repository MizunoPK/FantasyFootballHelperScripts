# Epic Request: Schedule Fetcher CLI Arguments and E2E Mode

**Context**: This is one of 7 runner script refactoring features originally designed in
KAI-10 (architectural_refactoring_configuration_management) but never implemented. The
full specification was completed in KAI-10 S2 and can be referenced directly.

**Existing spec**: feature-updates/KAI-10-architectural_refactoring_configuration_management/
feature_02_schedule_fetcher_cli/spec.md (all requirements, acceptance criteria, files to modify)

## Background

run_schedule_fetcher.py currently has minimal CLI argument support (only --enable-log-file).
Configuration is hardcoded: NFL_SEASON = 2025 is a module-level constant (line 31), and the
output path is hardcoded inline (line 56). The script has no E2E test mode and no --log-level
control.

The script already uses direct import (not subprocess) — no architecture migration needed.
ScheduleFetcher.py has zero config imports — simpler scope than the player fetcher.

## Goals

1. Add argparse to run_schedule_fetcher.py: --season, --output-path, plus universal args
2. Add universal args: --e2e-test, --log-level (NO separate --debug flag)
3. Remove NFL_SEASON module-level constant — replace with argparse default
4. Implement --e2e-test mode: fetches 1 week only, writes to /tmp, completes in ≤180 seconds
5. Wire --log-level to setup_logger() (currently hardcoded "INFO")

## E2E Mode

**Purpose:** The --e2e-test flag enables a repeatable, automated end-to-end scenario designed
for CI/CD integration testing. The integration test framework (feature 8 of this series) calls
each script with --e2e-test via subprocess to validate the full pipeline end-to-end without
needing human interaction. This is distinct from unit tests — it exercises the real network
call, real data parsing, and real file I/O through the actual CLI entry point.

**Requirements for a valid E2E mode:**
- Completes in ≤180 seconds (network latency must be within this budget)
- Writes all output to /tmp (never to data/) — /tmp is outside the repo and never committed
- Exits 0 on success
- Deterministic: same command always tests the same thing (week 1, fixed season)

**Behavior:** fetches 1 week of schedule data only (max_weeks=1), writes to
/tmp/schedule_e2e_test.csv. Does not write to data/season_schedule.csv.

## Design Decisions (from KAI-10 S2)

- No --data-folder: dropped as redundant for single-file output
- No --output-format: CSV-only output, no JSON option
- --request-timeout not exposed: internal constant (30.0s) stays internal
- --rate-limit-delay not exposed: internal constant (0.2s) stays internal
- No --debug flag: --e2e-test serves both E2E testing and debugging purposes
- E2E mode behavior: max_weeks=1 (requires adding max_weeks param to fetch_full_schedule())
- E2E output: writes to /tmp/schedule_e2e_test.csv (overrides --output-path default) to avoid
  contaminating data/season_schedule.csv
- Pattern: direct params (no @dataclass Settings) — runner is the main entry point
